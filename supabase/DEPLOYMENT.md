# Supabase Deployment Guide

This document explains how to deploy the Sales Arena Matcher system to Supabase, including the critical pg_cron setup for automatic weekly matching.

## Prerequisites

- Supabase account and project
- Supabase CLI installed (`npm install -g supabase`)
- Environment variables configured

## Deployment Steps

### 1. Set Environment Variables

Create a `.env.local` file in the project root (or configure in CI/CD):

```bash
VITE_SUPABASE_URL=https://YOUR_PROJECT_ID.supabase.co
VITE_SUPABASE_ANON_KEY=your_anon_key_here
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key_here
```

Get these values from Supabase Dashboard > Settings > API.

### 2. Deploy Database Migrations

```bash
supabase db push
```

This will:
- Create the `match_proposals` table
- Enable the `pg_cron` and `pg_net` extensions
- Create the cron job that triggers `weekly-matcher` hourly

**Troubleshooting:** If the migration fails because `pg_cron` is not available in your Supabase project, it may be using an older PostgreSQL version. Contact Supabase support to enable it, or manually create the cron job in the SQL editor.

### 3. Deploy Edge Functions

```bash
supabase functions deploy weekly-matcher --no-verify-jwt
```

**Important:** The `--no-verify-jwt` flag is critical because this function is called by pg_cron (which doesn't use auth tokens). The Edge Function validates room membership via database queries instead.

### 4. Verify pg_cron Setup

After deployment, verify the cron job was created:

1. Go to Supabase Dashboard > SQL Editor
2. Run: `SELECT * FROM cron.job WHERE jobname = 'weekly-matcher-hourly';`
3. Should show one row with a valid schedule

### 5. Test Match Generation

To manually trigger the weekly-matcher (useful for testing):

```bash
curl -X POST https://YOUR_PROJECT_ID.supabase.co/functions/v1/weekly-matcher
```

Should return: `{"ok":true,"week":"2026-07-14","created":{"room-id":N}}`

## How It Works

### Point-in-Time Matching (UTC-based)

The system creates match proposals when two users share the same UTC hour, regardless of their weekly affinity percentage.

**Example:**
- User A (Argentina, UTC-3) available: Monday 15:00 local → 18:00 UTC
- User B (Spain, UTC+1) available: Monday 17:00 local → 16:00 UTC
- **Result:** No match (different UTC hours)

But if User B is available at 20:00 local (18:00 UTC):
- **Result:** Match created ✅ (both at 18:00 UTC)

### Weekly Affinity vs Point-in-Time Matches

- **Affinity %:** Shows overlap across entire week (70% = 7 out of 10 hours overlap)
- **Match Proposals:** Created only when exact point-in-time UTC coincidences exist

The UI shows both:
1. **Heatmap (Afinidad Horaria tab):** Visual grid of point-in-time coincidences
2. **Match Cards (Panel de Control tab):** Actual proposals generated by weekly-matcher

### UTC Slot Calculation

The system uses **overlap logic** (not containment) for UTC slot calculation:

```javascript
if (s * 60 < endUtcMin && (s + 1) * 60 > startUtcMin) set.add(s);
```

This is critical for fractional timezone offsets like India (UTC+5:30). With containment logic, a 1-hour local block would never fit inside a 1-hour UTC slot, causing that user to have zero matches forever.

Overlap logic is used in 3 locations:
- `src/App.jsx` computeSlotSets() 
- `src/App.jsx` calculateEngine()
- `supabase/functions/weekly-matcher/index.ts`

**If you change this logic, update all 3 locations.**

## Troubleshooting

### Cron job not executing

Check execution logs:
```sql
SELECT * FROM cron.job_run_details WHERE job_id = (
  SELECT jobid FROM cron.job WHERE jobname = 'weekly-matcher-hourly'
) ORDER BY end_time DESC LIMIT 10;
```

### No match proposals being created

1. Verify `weekly-matcher` Edge Function is deployed
2. Check Edge Function logs: Supabase Dashboard > Logs > Edge Functions
3. Manually trigger: `curl -X POST https://YOUR_PROJECT_ID.supabase.co/functions/v1/weekly-matcher`
4. Verify availabilities are in the database: 
   ```sql
   SELECT room_id, COUNT(*) FROM availabilities GROUP BY room_id;
   ```

### Users don't see proposals in UI

- Proposals are only shown in "Panel de Control" tab
- Edge Function runs once per hour, so wait up to 60 minutes for new proposals
- Verify RLS policies allow user to see their own proposals:
   ```sql
   SELECT * FROM match_proposals WHERE member_a_email = 'user@example.com';
   ```
